Any text-bearing widget can embed a **value token** â€” a `:name:` placeholder that resolves to a live string at display time. Declare the source once at the top of your config; reference it anywhere in a `text =` field (or `top_text`, `bottom_text`, overlay text on image/gif widgets).

```
text = "Open 9â€“5  :clock.now:"
text = "Happy :date.today:!"
text = "Breathe. Move. Rise."    # no token â€” plain string
```

Tokens share the same `:slug:` syntax as [inline emoji](/assets/emoji/). An unknown `:slug:` renders as literal text (including the colons), so adding a `[[source]]` block is the only way to activate a token â€” a typo stays visible rather than silently disappearing.

## Declaring a source with `[[source]]`

Each value token needs a matching `[[source]]` block at the top level of your `config.toml` (alongside `[display]`, `[transitions]`, etc. â€” not nested inside `[[playlist.section]]`).

<TomlExample
  title="Clock token â€” 12-hour time with AM/PM"
  code={`[[source]]
id = "clock.now"          # the token name â†’ use :clock.now: in text fields
type = "clock"
format = "%-I:%M %p"      # strftime â€” "9:01 AM", "12:30 PM"
timezone = "America/New_York"   # IANA name; omit to follow the Pi's system clock`}
/>

<TomlExample
  title="Date token â€” day of week"
  code={`[[source]]
id = "date.today"
type = "date"
format = "%A"             # "Monday", "Tuesday", â€¦
timezone = "America/New_York"`}
/>

<TomlExample
  title="Static token â€” fixed text that never changes"
  code={`[[source]]
id = "brand.tagline"
type = "static"
value = "Breathe. Move. Rise."`}
/>

The `id` becomes the token name. It must match the pattern `[a-z_][a-z0-9_.]*` â€” lowercase letters, digits, underscores, and dots â€” to be usable as a `:token:`. An id that doesn't match the pattern is accepted, but the `:...:` placeholder won't be recognized at display time and renders as literal text instead. A dot in the name (like `clock.now`) is fine; it is just part of the name and has no special meaning.

## Using tokens in widgets

Reference a declared source by its `id`, wrapped in colons, in any `text =` field on a `message`, `two_row`, `gif`, or `image` widget:

<TomlExample
  title="Clock + date in a message widget"
  code={`[[source]]
id = "clock.now"
type = "clock"
format = "%-I:%M %p"
timezone = "America/New_York"

[[source]]
id = "date.today"
type = "date"
format = "%A"
timezone = "America/New_York"

[[playlist.section]]
mode = "slideshow"
hold_time = 10

[[playlist.section.widget]]
type = "message"
text = ":date.today: :clock.now:"
font_color = [255, 183, 3]`}
/>

Tokens compose with everything else on the widget â€” `font_color = "rainbow"`, `animation = "typewriter"`, `border = "rainbow"`, inline emoji. The token is substituted before layout, so the rest of the pipeline sees a plain string.

## Token color

A `[[source]]` block can carry its own `color`. When it does, the value substituted for its `:id:` token renders in that color, while the literal text around it keeps rendering in the widget's own `font_color` â€” the two compose in one line.

<TomlExample
  title="A colored token composed with a plain font_color"
  code={`[[source]]
id = "studio.status"
type = "static"
value = "OPEN"
color = [0, 220, 90]          # green â€” colors the token only

[[playlist.section.widget]]
type = "message"
text = "Firebird Yoga: :studio.status:"
font_color = [80, 140, 255] # blue â€” colors everything else`}
/>

`Firebird Yoga: ` renders blue (the widget's `font_color`); `OPEN` renders green (the source's `color`). `color` accepts the same forms as `font_color`: a constant `[r, g, b]`, a string shorthand (`"rainbow"`, `"color_cycle"`, `"shimmer"`), or an inline table (`{style = "gradient", from = [...], to = [...]}`, or a plugin-contributed style such as `{style = "stocks.trend", symbol = "AAPL"}`). A source with no `color` field behaves exactly as before â€” its token inherits whatever color the host widget is already using.

Token colors compose with a per-character host color too: a message with `font_color = "rainbow"` and a colored token keeps sweeping the rainbow across its literal text while the token itself stays fixed in the source's color.

**This release: `message` widgets only.** `two_row` and the text overlay on `gif`/`image` widgets don't read a source's `color` yet â€” a token there always renders in the widget's own color, regardless of what the source declares. Support for those is a planned follow-up.

Two honest scope boundaries, both narrow:

- **A `flair.fisheye` lens drops token colors.** Under [`animation = "flair.fisheye"`](/concepts/animations/#flairfisheye-flair-pack), the whole message renders in the widget's `font_color` â€” a source's `color` isn't applied while the text is passing through the lens.
- **A `typewriter` reveal with inline emoji delays token color until fully revealed.** With `animation = "typewriter"`, a message that mixes a colored token with a `:slug:` emoji renders the token in the host's `font_color` while the reveal is in progress, then switches to the source's color once the whole message is on screen. (A typewriter reveal with no inline emoji colors the token correctly the whole time.)

## The three source types

### `clock` â€” live wall-clock time

`format` is a [Python `strftime` format string](https://docs.python.org/3/library/datetime.html#strftime-and-strptime-format-codes). Common patterns:

| Format        | Output                        |
| ------------- | ----------------------------- |
| `"%-I:%M %p"` | `9:01 AM`                     |
| `"%I:%M %p"`  | `09:01 AM` (zero-padded hour) |
| `"%H:%M"`     | `21:01` (24-hour)             |
| `"%I:%M"`     | `09:01` (12-hour, no AM/PM)   |

`timezone` is an IANA timezone name (e.g. `"America/New_York"`, `"Europe/London"`, `"Asia/Tokyo"`). Omit it and the token follows the Pi's system clock â€” a fresh Pi is often UTC, so set the timezone explicitly.

### `date` â€” calendar date

Same as `clock` but formatted for dates. Common patterns:

| Format        | Output       |
| ------------- | ------------ |
| `"%A"`        | `Monday`     |
| `"%B %-d"`    | `June 30`    |
| `"%m/%d"`     | `06/30`      |
| `"%a %-d %b"` | `Mon 30 Jun` |

`timezone` works the same as for `clock`.

### `static` â€” fixed text

`value` is a plain string that never changes. It is the simplest source and a good way to keep a reusable phrase in one place and reference it from multiple widgets.

```toml
[[source]]
id = "studio.name"
type = "static"
value = "Firebird Yoga"

# then in any widget:
# text = ":studio.name: Â· Breathe. Move. Rise."
```

## How live updates work

The display engine refreshes all synchronous sources (clock, date, static) once per second. A source's value is compared to the previous tick â€” if it changed, the internal version counter increments and any widget that references that source rebuilds its display string and recalculates its layout. Widgets with no tokens are untouched.

**Constant-width formats** (`%H:%M`, `%I:%M %p`) produce a value the same pixel width every tick, so the panel never reflowing mid-hold. **Variable-width formats** (`%A` â€” "Monday" vs "Saturday") reflow at most once per value change (per minute or per day), cleanly at the widget's next hold tick.

During scrolling, transitions, and typewriter reveals, resolution is frozen â€” if a value changes mid-scroll, the new value takes effect at the next hold rather than interrupting the in-flight animation.

## Resolution order: emoji wins, then source, then literal

If a `:slug:` matches an emoji in the [emoji registry](/assets/emoji/), it renders as a pixel-art sprite â€” the emoji takes priority over any source with the same name. The validator (`make validate`) rejects a `[[source]]` `id` equal to an existing emoji slug, so this collision cannot occur in a valid config.

An `:unknown:` slug that matches neither an emoji nor a declared source renders as literal text (`:unknown:`). This is intentional â€” a missing or misspelled source name stays visible so you can spot it at a glance.

## Live (polled) sources

The `clock`, `date`, and `static` sources compute their value locally and instantly â€” they need no network and never fail. Some source types work differently: they fetch data in the background on a repeating `interval` (in seconds) and push the result into the token when the fetch completes.

These **polled sources** are contributed by plugins. A plugin registers a source type via `api.source` by subclassing `PolledDataSource` and implementing an `async def update()` method that calls `self._set_value(...)` with a new string. The engine spawns a supervised background task that calls `update()` every `interval` seconds for as long as the display is running.

A polled `[[source]]` block looks the same as a built-in one â€” the plugin type name and any type-specific fields sit alongside `id` and `interval`:

```toml
[[source]]
id       = "my.token"
type     = "some.plugin.type"   # contributed by a plugin
interval = 300                  # fetch every 5 minutes
# ...type-specific fields...
```

**Before the first fetch completes**, the token shows a short placeholder (e.g. `â€¦`) so your layout is never blank. Once the first value arrives, it replaces the placeholder and the token updates live on the same reflow path as clock and date â€” any widget that references it rebuilds its display string and recalculates layout automatically.

If a fetch fails, the previous value is kept and the task retries on the next `interval`. A crashed fetch task is supervised and restarted, so a transient network error doesn't freeze the token permanently. If a polled source like `:weather.nyc:` stops updating, the [web UI's Monitors panel](/concepts/web-status-ui/#status-tab--monitors) shows whether it is erroring (with the error message and a retry countdown) or stale (fetch wedged, no response).

### Example: live weather

The [weather plugin](/plugins/available/#weather) contributes a `weather.current` source. Declare it once:

```toml
[[source]]

type     = "weather.current"         # from the weather plugin
location = "New York, US"            # name / ZIP / "lat,lon"
interval = 1800                      # refresh every 30 minutes
format   = "{temp_f}Â°F {condition}"  # optional; this is the default
```

Then reference it in any widget's text:

```toml
[[playlist.section.widget]]

```

The `format` string interpolates the current-conditions fields `temp_f`, `temp_c`, `condition`, `feelslike_f`, `feelslike_c`, `humidity`, `wind_mph`, and `emoji`. The `emoji` field expands to a condition icon that renders as a sprite, so `format = "{temp_f}Â° {emoji}"` gives `72Â° â˜€`. The API key comes from `WEATHERAPI_KEY` in your `.env` (never in config); see the [weather plugin](/plugins/available/#weather) for setup.

## Tips

### Zero-pad the hour to avoid reflowing

`%H:%M` (24-hour, `09:01`) and `%I:%M` (12-hour with leading zero, `09:01`) keep the time string a fixed pixel width. `%-I:%M` (no leading zero, `9:01`) gives a narrower string for single-digit hours â€” fine in `slideshow` hold, but the message will reflow at 10:00 when the hour gains a digit.

### Multiple `[[source]]` blocks are fine

Each block declares one source. Define as many as you need; they are all global across every section.

### Tokens work in `ticker` and `one_at_a_time` modes too

`ticker` and `one_at_a_time` sections scroll widgets continuously. Tokens resolve normally â€” the value used is whatever the source holds at the moment the widget is drawn. A per-minute date or AM/PM flip takes effect the next time the widget passes through the visible area.
