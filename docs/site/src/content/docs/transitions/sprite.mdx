The flair pack (`led-ticker-flair`) ships two categories of transition: **homage sprite-trails** — pixel-art characters that cross the panel — and **text-effect transitions** that manipulate the outgoing content itself. Both are referenced by their namespaced slug, e.g. `transition = "pokeball.forward"` or `transition = "flair.spinout"`.

## Homage transitions (flair pack)

The four homage families — **nyancat**, **pokeball**, **pacman**, and **sailor_moon** — ship together in the `led-ticker-flair` package (one install, four entry points, each with `.forward`, `.reverse`, and `.alternating` variants).

### Install

Add one line to `config/requirements-plugins.txt` (copied from `config/requirements-plugins.example.txt`):

```text
led-ticker-flair
```

One install registers all four families. See [Plugins](/plugins/) for the install flow.

A bare `transition = "nyancat"` (the pre-plugin spelling) raises a config-load error pointing at the namespaced `nyancat.forward` slug — the namespaced form is what the flair pack registers.

:::note[Existing configs need no change]
The type strings (`nyancat.forward`, `pokeball.alternating`, etc.) are unchanged — only the install source consolidates. Configs already using these transitions keep working as-is. If you previously pinned individual family tags (e.g. `@nyancat-v0.1.0`), those installs continue to work; switch to the single flair line at your convenience.
:::

## Hires

`nyancat.forward` and `pokeball.forward` auto-activate hi-res sprites whenever the canvas is at `default_scale > 1` — that is, any big / scaled sign (see [Display](/concepts/display/)), typically a bigsign-style build. The hi-res variants are noticeably more detailed and read better at distance:

- **nyancat** — animated webp Nyan Cat sprite at native physical resolution
- **pokeball** — rolling pokeball with animated run-cycle sprite

The `sailor_moon` and `pacman` families keep their 8-bit look at all scales — the chunky aesthetic is part of the design.

## Other plugin sprites

The rolling-**baseball** family (`baseball.roll`, `baseball.roll_reverse`, `baseball.roll_alternating`) ships with the **[baseball](https://github.com/JamesAwesome/led-ticker-plugins/tree/main/plugins/baseball)** package in the led-ticker-plugins monorepo, alongside that package's MLB widgets and `:baseball.ball:` emoji. Install it via `config/requirements-plugins.txt` (see [Plugins](/plugins/)) to use it.

## Text-effect transitions (flair pack)

### `flair.spinout` (flair >= 0.3.0, core >= 4.6)

`flair.spinout` spins the outgoing widget's content out of view — starting at rest, then accelerating — before cutting to the incoming widget. Unlike a sprite trail, nothing crosses the panel; the content itself rotates and disappears. Transition duration follows `[transitions] duration` (default 0.5 s; sprite-style durations of 1.0–1.5 s look best for a full revolution or two).

<DemoGif
  src="/demos-pinned/transition-flair-spinout.gif"
  caption="`flair.spinout` — outgoing text accelerates and spins out before the incoming widget appears"
/>

#### Install

`flair.spinout` is part of the same `led-ticker-flair` package as the homage sprite-trails:

```text
led-ticker-flair
```

One install registers all flair transitions and animations. See [Plugins](/plugins/) for the install flow.

#### Minimal config

<TomlExample
  title="Minimal"
  code={`[transitions]
default = "flair.spinout"
duration = 1.0

[[playlist.section]]
mode = "slideshow"
hold_time = 3.0

[[playlist.section.widget]]
type = "message"
text = "SPIN OUT"
center = true
font_color = [0, 220, 255]

[[playlist.section]]
mode = "slideshow"
hold_time = 2.5

[[playlist.section.widget]]
type = "message"
text = "AND CUT"
center = true
font_color = [80, 255, 80]`}
/>

#### Knobs

| Knob          | Default | Notes                                                   |
| ------------- | ------- | ------------------------------------------------------- |
| `revolutions` | `2`     | Integer >= 1. Full turns before the content disappears. |
| `direction`   | `"cw"`  | `"cw"` (clockwise) or `"ccw"` (counter-clockwise).      |

Set knobs via an inline table on `[transitions] default` or on the section's `transition` field:

```toml
[transitions]
default = {type = "flair.spinout", revolutions = 1, direction = "ccw"}
duration = 1.0
```

#### Behavior notes

- **Background color holds.** The outgoing section's `bg_color` fills the panel for the entire spin duration — the background never flashes or drops to black mid-transition.
- **Scrolled text spins from its parked position.** If the outgoing widget had been scrolling, `flair.spinout` picks up at the position where the text stopped (the same `outgoing_scroll_pos` continuity used by the push family).
- **Colors freeze during the spin.** Animated `font_color` providers (`rainbow`, `color_cycle`, `shimmer`) are paused for the outgoing widget while it spins, consistent with how transitions pause frame-aware effects in general.
- **Pairs with `flair.propeller`.** Use `flair.spinout` as the inter-widget or between-sections transition alongside `animation = "flair.propeller"` on `message` widgets for a full spin-in / spin-out cycle. See [flair.propeller → Pairing with flair.spinout](/concepts/animations/#pairing-with-flairspinout).

### `flair.fireworks` (flair >= 0.7.0, core >= 4.10)

`flair.fireworks` holds the outgoing widget while staggered firework bursts launch as rising streaks from the bottom edge, then open into black "burn-through" holes ringed by radial spoke trails — an asterisk of bright white heads and fading colored tails, not a ring outline. Past the halfway point every burst's hole blooms in lockstep toward full-panel coverage, revealing the incoming widget through the expanding circles and blacking out everything outside their union, until embers fade and the whole panel cuts to the incoming section. Transition duration follows `[transitions] duration` — about 1.2 s gives the bursts room to stagger, open, and bloom without feeling rushed.

<DemoGif
  src="/demos-long/transitions-fireworks.gif"
  caption="`flair.fireworks` — staggered bursts open over the outgoing widget, then bloom to reveal the incoming widget"
/>

#### Install

`flair.fireworks` is part of the same `led-ticker-flair` package as the homage sprite-trails and `flair.spinout`:

```text
led-ticker-flair
```

One install registers all flair transitions and animations. See [Plugins](/plugins/) for the install flow.

#### Minimal config

<TomlExample
  title="Minimal"
  code={`[transitions]
between_sections = "flair.fireworks"
duration = 1.2`}
/>

With knobs, set via an inline table on the section's `transition` field (or on `[transitions] default`/`between_sections`):

<TomlExample
  title="With knobs"
  code={`[[playlist.section]]
transition = {type = "flair.fireworks", bursts = 3, colors = [[255, 60, 60], [0, 220, 220]]}`}
/>

#### Knobs

| Knob     | Default                  | Notes                                                                                                                                                                                                                                                                                                                                                                                           |
| -------- | ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `bursts` | size-adaptive (3-6)      | Number of firework bursts. Auto-picked from the panel's width-to-height ratio when omitted — wider panels get more bursts (smallsign: 5, bigsign: 4, longboi: 6) — or set explicitly in `[2, 8]`.                                                                                                                                                                                               |
| `colors` | 8-color built-in palette | List of `[r, g, b]` triples, cycled across bursts in order. From core **4.11** the standard `transition_colors` field works too (it reaches any plugin transition that declares a `colors` parameter, in per-section `transition =` and table-form `between_sections` — the same places built-in transition colors apply); a `colors` key written directly in the transition table always wins. |
| `seed`   | `None` (OS entropy)      | Fixes the RNG for reproducible bursts — useful for regression tests and reproducible demo GIFs. Not a promise that the same seed plans the same sequence forever: a re-fire of the same running instance continues drawing from that same RNG stream, so consecutive firings on a live sign still vary. Leave unset for a real deploy.                                                          |

#### Behavior notes

- **Size-adaptive by default.** Burst count, radius, and stagger timing are all derived from the panel's actual width/height at draw time — the same `transition = "flair.fireworks"` line looks right on smallsign, bigsign, and longboi without per-sign tuning.
- **Two-phase design, painted through black.** There is no `GetPixel` on real hardware (the framebuffer stores pre-computed GPIO bitplane data, not readable RGB), so nothing can be composited from a previous frame — every frame is drawn from scratch from the widgets plus burst geometry. Phase one (`t < 0.5`) plays over the outgoing widget; phase two (`t >= 0.5`) switches to the incoming widget and blacks out everything outside the union of the (still-growing) burst circles, guaranteeing full coverage by `t == 1.0` regardless of any individual burst's stagger.

### `flair.stickers` (flair >= 0.9.0, core >= 4.18)

`flair.stickers` sticker-bombs the panel: emoji "stickers" pop on one at a time in a random order over the outgoing widget until they fully cover it at the halfway point, then pop off in an independent random order, revealing the incoming widget in the gaps. Each sticker sits on a tilted die-cut card so the wall reads as stickers on glass. Pass `emoji = ["taco"]` for a themed wall of one emoji, a list for a mix, or omit it for a random assortment. Since core 4.21.0 the drawable set includes the ~1,360-emoji [standard-emoji pack](/assets/emoji/#standard-emoji), so random mode (flair >= 0.13.0) samples up to **12 distinct emoji per firing** and builds the wall from those — wildly different stickers every firing, a coherent wall within one.

<DemoGif
  src="/demos-pinned/transition-flair-stickers.gif"
  caption="`flair.stickers` — a wall of taco cards buries the outgoing widget, then peels away to reveal the incoming one"
/>

#### Install

`flair.stickers` ships in the same `led-ticker-flair` package as the sprite-trails and the other text-effect transitions:

```text
led-ticker-flair
```

One install registers every flair transition and animation. See [Plugins](/plugins/) for the install flow.

#### Minimal config

<TomlExample
  title="Themed taco wall"
  code={`[[playlist.section]]
mode = "slideshow"
hold_time = 3.0
transition = {type = "flair.stickers", emoji = ["taco"]}

[[playlist.section.widget]]
type = "message"
text = "TACO"
center = true
font_color = [255, 190, 0]`}
/>

#### Knobs

| Knob      | Default                     | Notes                                                                                                                                                                                                                                                                                                                                                          |
| --------- | --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `emoji`   | omitted (random assortment) | List of emoji slugs (no colons), e.g. `["taco"]` or `["sun", "moon", "star_yellow"]`. A single-item list is a themed wall; a list mixes (never capped); omitting it samples ≤ 12 distinct slugs per firing from the full drawable set (~1,360 with the standard-emoji pack) and draws each sticker from those. An unknown slug fails at `led-ticker validate`. |
| `backing` | `"card"`                    | Sticker body. `"card"`: tilted black card with a white rim — the only mode that fully covers the panel at the midpoint. `"shadow"`: black silhouette halo, no card. `"none"`: bare sprites. Shadow/none read as an emoji **swarm** — gaps stay visible, by design.                                                                                             |
| `seed`    | `None` (OS entropy)         | Fixes sticker positions, angles, and pop-on/off order for reproducible runs. Leave unset for a real deploy (every firing varies).                                                                                                                                                                                                                              |

#### Behavior notes

- **The card is what guarantees full coverage.** Each `"card"` sticker is squared to its grid cell, so a jittered grid of cards covers the panel completely at the midpoint regardless of the emoji's shape. `"shadow"` and `"none"` deliberately trade that guarantee for a lighter, see-through swarm.
- **Only real slugs.** A typo like `emoji = ["fire"]` (there is no `fire` slug) fails at config-load with the bad slug named — see [Emoji](/assets/emoji/) for the full list.
- **Random mode is chaos across firings, coherence within one.** With no `emoji`, each firing samples up to 12 distinct slugs from the full drawable set and builds the wall from those — one firing might be lobsters, dice, and hearts; the next guitars, planets, and pizza. Explicit `emoji = [...]` lists are never capped.

### `flair.poker` (flair >= 0.10.0, core >= 4.18)

`flair.poker` deals the panel in with card suits. Rainbow ♥ ♦ ♣ ♠ glyphs pattern in over the outgoing widget, then each one emits **suit-shaped** rainbow ripple rings — an expanding heart emits heart-shaped rings, a club club-shaped ones — that wash the incoming widget in until the panel is covered. Pass `suits = ["diamonds"]` (or `["clubs"]`, etc.) for a single suit, or omit it for all four.

  caption="`flair.poker` — rainbow suit glyphs emit suit-shaped ripples that wash in the incoming widget"
/>

#### Install

`flair.poker` ships in the same `led-ticker-flair` package:

```text
led-ticker-flair
```

One install registers every flair transition and animation. See [Plugins](/plugins/) for the install flow.

#### Minimal config

<TomlExample
  title="All four suits"
  code={`[[playlist.section]]
mode = "slideshow"
hold_time = 3.0
transition = "flair.poker"

[[playlist.section.widget]]
type = "message"
text = "ANTE UP"
center = true
font_color = [255, 255, 255]`}
/>

#### Knobs

| Knob    | Default             | Notes                                                                                                                                                  |
| ------- | ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `suits` | all four            | List from `hearts` / `diamonds` / `clubs` / `spades`, e.g. `["diamonds"]` for a single-suit wash. An unknown suit name fails at `led-ticker validate`. |
| `seed`  | `None` (OS entropy) | Fixes the glyph layout and ripple timing for reproducible runs. Leave unset for a real deploy.                                                         |

#### Behavior notes

- **The reveal is against black.** Like `flair.fireworks`, the second half reveals the incoming widget inside the ripple wakes against black (hardware can't composite two widgets — there is no `GetPixel`), then covers the whole panel by the end. The suit glyphs and their rings share one shape function, which is what lets the ripples take each suit's outline.
- **Single-suit is a theme, all-suits is a shuffle.** `suits = ["hearts"]` gives a wall of heart ripples; omitting `suits` mixes all four across the grid.

#### Knobs

| Knob    | Default                               | Notes                                                                                                         |
| ------- | ------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| `seed`  | `None` (OS entropy)                   | Fixes the bolt path for reproducible strikes. Leave unset for a real deploy — every firing gets a fresh bolt. |
| `color` | electric blue-white `[150, 190, 255]` | Tints the bolt trail and the crack edges as they open. The strike head is always white.                       |

#### Behavior notes

- **Two phases.** The first phase draws the bolt left→right over the outgoing widget; then it cuts to the incoming widget and opens the crack, blacking out everything outside the widening zigzag gap. **The incoming is revealed against black, not composited over the old content** (hardware has no `GetPixel`, so the outgoing's pixels can't be read back and slid apart). Section backgrounds (`bg_color`) show inside the gap from the first sliver.
- **No warm-up.** Every frame is a pure function of the transition's progress; the first firing after boot costs the same as every other.

### `flair.fairy` (flair >= 0.12.0, core >= 4.18)

`flair.fairy` sends a Tinkerbell-style fairy — a white-hot sparkle-star — across the panel trailing a long cone of gold pixie dust and settling a thin line; when the fairy reaches the far edge the line opens into two twinkling gold edges, revealing the incoming widget in the gap. It comes in three direction variants: `flair.fairy.forward` (left→right), `flair.fairy.reverse` (right→left), and `flair.fairy.alternating` (flips every firing).

#### Knobs

| Knob    | Default                | Notes                                                                                                                   |
| ------- | ---------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| `seed`  | `None` (OS entropy)    | Fixes the flight path and the pixie-dust spark field. Leave unset for a real deploy — every firing gets a fresh flight. |
| `color` | gold `[255, 215, 120]` | Tints the pixie dust, the settled line, and the edges as the line opens. The fairy head is always white.                |

Direction is **not** a knob — pick the variant (`.forward` / `.reverse` / `.alternating`).

#### Behavior notes

- **Two phases.** The first half flies the fairy across, drawing a dwelling cone of dust over the outgoing widget; the second half opens the settled line to reveal the incoming widget against black outside the gap (same `GetPixel` constraint as `flair.lightning` and `flair.poker`). Section `bg_color` shows inside the gap from the first sliver.
- **Near-straight flight.** The path is a gentle center-band line — the magic is in the dust, not a loop-de-loop — so the settled line never doubles back on itself and can split cleanly.
- **Stateless sparks, no warm-up.** Every spark is a pure function of the column and the transition's progress — no particle simulation, no caches — so the first firing after boot costs the same as every other.
