The `rss_feed` widget fetches a remote RSS feed and turns each headline into its own `TickerMessage` that scrolls in the section. The feed title appears first, followed by up to `max_stories` stories, each with its own color. Updates happen in the background via async polling with exponential backoff — the display keeps running even if the feed is temporarily unreachable.

## Common patterns

### Uniform color for all stories — rainbow sweep across every headline

<TomlExample
  code={`[[playlist.section.widget]]
type = "rss_feed"
feed_url = "https://www.nintendolife.com/feeds/news"
font_color = "rainbow"`}
/>

### Cap headlines to the top 3 stories with a slower poll

<TomlExample
  code={`[[playlist.section.widget]]
type = "rss_feed"
feed_url = "https://feeds.bbci.co.uk/news/rss.xml"
max_stories = 3
update_interval = 3600`}
/>

### Mix two feeds in one section by listing both widgets — stories from each feed interleave naturally as the section loops

<TomlExample
  code={`[[playlist.section.widget]]
type = "rss_feed"

max_stories = 3

[[playlist.section.widget]]
type = "rss_feed"
feed_url = "https://feeds.bbci.co.uk/news/rss.xml"
max_stories = 3`}
/>

## Tips

### Network latency on first appearance

The widget performs an initial fetch at startup before entering the section. Feeds with poor uptime or slow servers can stall the first appearance until the fetch completes or the retry backoff exhausts. Choose reliable feed URLs; the widget will display nothing until the first successful fetch returns.

### Be polite with poll intervals

The default `update_interval` of 1800 seconds (30 minutes) is a reasonable balance between freshness and server load. Dropping below 60 seconds on a public feed is considered abusive and may get your IP rate-limited or blocked. Raise the interval for high-traffic or third-party feeds.
