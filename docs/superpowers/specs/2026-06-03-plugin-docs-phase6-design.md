# Phase 6 — Plugin Documentation — Design

**Date:** 2026-06-03
**Status:** Approved (brainstorm), pending implementation plan
**Context:** Final phase of the pool extraction (`docs/superpowers/plans/2026-06-03-pool-plugin-extraction.md`). Phases 1–5 are merged + hardware-validated; the declarative plugin-requirements file shipped in PR #152.

## Goal

Document the plugin system now that pool is an external plugin, and make the
`led-ticker-pool` repo self-documenting. Two parts:

1. **led-ticker** — a docs-site **Plugins** overview page (install + concepts +
   pointer), a CLAUDE.md plugin-invariants section, and a config.example note.
2. **led-ticker-pool** — move the canonical pool widget docs into the repo's
   `README.md`, including the rendered GIF screenshots.

The deep **plugin authoring guide** is explicitly a **future phase** (not this one).

## Decisions (from brainstorm)

- **Canonical home for pool docs = the `led-ticker-pool` README.** The existing
  docs-site `widgets/pool.mdx` is slimmed to a pointer (preserving the
  `/widgets/pool/` URL); no duplicated full content that can drift.
- **Screenshots = the existing rendered GIF demos** (`widget-pool.gif`,
  `widget-pool-two-row.gif`), copied into the led-ticker-pool repo. No new
  hardware photos.
- **Docs-site Plugins page = lean overview** (Approach A): what plugins are,
  how install works, the `[plugins]` relationship, a short authoring teaser that
  links to the existing `docs/plugin-system.md` engineering reference, and an
  "Available plugins" list. No full authoring guide (future phase).

## Design

### Part 1 — `led-ticker-pool/README.md` (canonical pool docs)

Expand from the scaffold into the full widget documentation. Sections:

1. **Title + one-line intro** — pool water-temperature widget
   (`type = "pool.monitor"`), backed by an InfluxDB v2 server.
2. **Screenshots** — embed `docs/widget-pool.gif` (ticker) and
   `docs/widget-pool-two-row.gif` (two_row), copied from led-ticker's
   `docs/site/public/demos-long/`. Each with a caption naming the layout.
3. **Install** — (a) via led-ticker's `config/requirements-plugins.txt`
   (`cp` the example, rebuild); (b) the Docker auto-install; (c) standalone
   `pip install "git+https://github.com/JamesAwesome/led-ticker-pool.git@main"`.
   Note entry-point auto-registration (no `[plugins]` change needed).
4. **Configuration** — the full options table (relocated/copied from
   `docs/content-source/widgets/pool.md`: title, sensor_id, units,
   update_interval, current_window, stale_after, influxdb_url/org/bucket/token,
   layout, top_font*/bottom_font*/top_row_height, label_color, font/font_size).
5. **Layouts** — `ticker` (default; single-row segmented screens + trend arrow)
   vs `two_row` (title + today + 7-day + season HI + season LO, label-on-top /
   big-number-on-bottom). Pulled from the CLAUDE.md pool note + the old pool.mdx.
6. **Example config** — a `ticker` snippet and a `two_row` snippet
   (`[[playlist.section.widget]]` blocks).
7. **InfluxDB setup** — the four env vars (`INFLUXDB_URL/ORG/BUCKET/TOKEN`) and
   what measurement/data the widget queries (Flux over HTTP).
8. **Development** — keep/refine the existing section (editable install from a
   sibling led-ticker checkout, `uv run pytest`, the rgbmatrix-stub pythonpath note).
9. **Links** — back to the led-ticker project + its docs-site Plugins page.

GIF assets: copy the two files into `led-ticker-pool/docs/` and reference them
relatively (`docs/widget-pool.gif`) so they render on GitHub.

### Part 2 — led-ticker docs site

1. **New `plugins/` overview page** (`docs/site/src/content/docs/plugins/index.mdx`
   or `plugins.mdx`):
   - What plugins are (extend led-ticker — widgets, transitions, emojis, fonts,
     lifecycle hooks — without forking core).
   - **Installing a plugin:** `config/requirements-plugins.txt` (copy from the
     tracked `.example`), constraint-based build install, Docker/bare-metal
     auto-install, entry-point auto-registration (an installed plugin registers
     itself — no `[plugins]` block required).
   - **Relationship to `[plugins]`:** controls *loading* (`enabled`, `disable`,
     local `dir`), not *installing* — complementary.
   - **Writing a plugin (teaser):** one short paragraph + link to
     `docs/plugin-system.md`; note a full authoring guide is a future addition.
   - **Available plugins:** a list with **pool** → the `led-ticker-pool` README
     (canonical).
2. **Reconcile `widgets/pool.mdx`:** slim it to a pointer page — "pool is now the
   external `led-ticker-pool` plugin; full docs: [repo README]; install:
   [Plugins page]." Preserves the `/widgets/pool/` URL + Widgets-nav discoverability.
3. **Nav (`astro.config.mjs`):** add a **Plugins** sidebar group (the overview
   page; pool listed there pointing to the repo). Keep the slimmed pool pointer
   reachable. Leave `docs/content-source/widgets/pool.md` untouched.

### Part 3 — CLAUDE.md + config note

1. **CLAUDE.md** — add a concise **"Plugin invariants"** section (agent-facing):
   public surface is `led_ticker.plugin` only; `led_ticker.plugins` entry-point
   registration via `register(api)`; `API_VERSION`; install path is
   `config/requirements-plugins.txt` (constraint-based, not `--no-deps`);
   `validate_config` convention; error isolation; pool lives in `led-ticker-pool`.
   Link to `docs/plugin-system.md` for depth. Keep it short.
2. **config.example** — refine the existing pool install comment to point at the
   docs-site Plugins page / `requirements-plugins.txt`. No `[plugins]` block
   (auto-discovery).

### Cross-linking

- pool README → led-ticker project + Plugins page.
- Plugins page → pool README (canonical) + `docs/plugin-system.md`.
- `widgets/pool.mdx` stub → pool README + Plugins page.

## Logistics

Two repos → **two PRs**:
- **PR (led-ticker-pool):** Part 1 — README rewrite + copied GIFs.
- **PR (led-ticker):** Parts 2–3 — Plugins page, pool.mdx stub, nav, CLAUDE.md,
  config note.

Order: land the pool README first (so the led-ticker Plugins page can link to a
real canonical page), or land together and fix links. Either works; the plan
will build the README PR first.
