# Web status UI v1.1 — design

**Date:** 2026-06-11
**Status:** approved scope, pre-implementation
**Builds on:** `docs/superpowers/specs/2026-06-10-web-status-ui-design.md` (v1, shipped in
PR #192). All v1 invariants carry forward unchanged: read-only, always-redact,
never-500 degraded states, rgbmatrix import purity, no write path to the display.

## Summary

Two usability additions to the sidecar, scoped with James on 2026-06-11:

1. **Validate tab works on real files** — a dropdown of the config directory's
   `.toml` files with server-side validation (exact results, secrets never
   transit), plus a "load redacted copy into the editor" convenience with a
   caveat banner.
2. **Inventory tab** — what this sign has to work with: fonts, image/gif
   assets, emoji slugs, and the real names each plugin registered.

## Component 0: `[web]` key rename — `http_host` / `http_port`

Rename `host` → `http_host` and `port` → `http_port` (TOML keys AND the
`WebConfig` dataclass fields — the drift test enforces 1:1) to match the
`[busy_light]` convention (`http_host` / `http_port`). `token` and
`status_path` are unchanged. The old keys raise a clear `ValueError` at
config load naming the rename ("web.port was renamed to web.http_port").
No auto-fix machinery: the field shipped one day ago with a single install;
a load-time error with the fix in the message is proportionate. Deploy note:
longboi's `config.toml` needs the rename when it picks up this release.

## Component 1: validate-by-file

### `GET /api/configs`

Lists candidate config files: the basenames of `*.toml` directly inside the
config directory (`config_path.parent`), non-recursive, sorted. Response:

```json
{"configs": ["config.toml", "config.bigsign.example.toml"], "running": "config.toml"}
```

`running` is the basename of the live config so the UI can mark it.

### `POST /api/validate-file`

Body: `{"name": "config.bigsign.example.toml"}`. Validates that file
server-side via the existing path-based `validate_config` and returns the same
issues JSON as `POST /api/validate` (`_result_to_json` — still excludes
`result.path`).

**Path-traversal guard — the security surface of this feature.** The
`name` must survive ALL of:

1. `name == Path(name).name` (basename only — rejects `../x`, `a/b`, absolute paths)
2. The resolved candidate (`(config_dir / name).resolve()`) is a file whose
   parent is exactly `config_dir.resolve()` (defeats symlink hops out of the dir)
3. The name appears in the same listing `GET /api/configs` would produce
   (suffix check included — only `.toml`)

Violations return `404 {"error": "unknown config"}` — deliberately the same
response as a genuinely absent file, so the endpoint is not a filesystem
oracle. Tests must cover: `../escape.toml`, `/etc/passwd`, `sub/dir.toml`,
`x.txt`, a symlink inside config-dir pointing outside it, and the happy path.

### Validate tab UI

Above the existing textarea: a `<select>` of `/api/configs` entries (running
config marked "· running") with two buttons:

- **Validate file** → `POST /api/validate-file`, results render in the
  existing issues pane. Exact: validates the real file with real values.
- **Load redacted copy** → fetches a NEW endpoint variant
  `GET /api/config?name=<file>` (the existing `/api/config` handler gains an
  optional `name` query param, guarded by the same three traversal checks;
  no param = the running config, unchanged behavior) and fills the textarea
  with the redacted TOML. A persistent caveat banner appears above the
  editor: *"Loaded copy is redacted — `•••` values may validate differently
  than the real file. Use Validate file for exact results."*

## Component 2: Inventory tab

### `GET /api/inventory`

Filesystem-and-registry listing, assembled per request (no caching — matches
the config handler's read-per-request posture). Shape:

```json
{
  "fonts": {
    "user": ["my-font.ttf"],
    "bundled": ["4x6", "5x8", "6x12", "...", "hires/NotoSans.ttf"]
  },
  "assets": [{"path": "gifs/cat.gif", "bytes": 12345}],
  "emoji": {"core": ["star", "flower", "..."], "hires_only": ["moon"]}
}
```

- **fonts.user** — filenames in `<config dir>/fonts/` (the dir
  `app._configure_user_font_dir` anchors to). Missing dir → `[]`.
- **fonts.bundled** — BDF alias names plus hires font filenames from the
  installed package (`led_ticker/fonts/`). Implementation may enumerate via
  plain filesystem/`importlib.resources` listing — do NOT import the font
  loader if doing so threatens the rgbmatrix purity tripwire; verify either
  way (the purity test is the arbiter).
- **assets** — recursive listing under the config dir of
  `.gif/.png/.jpg/.jpeg/.webp/.bmp` files, paths relative to the config dir,
  with sizes. Bounded: stop at 500 entries and set `"assets_truncated": true`
  (a config dir is user-controlled; don't walk forever).
- **emoji.core / hires_only** — from `EMOJI_REGISTRY` / `HIRES_REGISTRY`
  (core registries; verify import purity). Plugin emoji are NOT visible to
  the sidecar (it doesn't load plugins) — they come from Component 3 via
  status.json instead, and the UI merges the two sources.

### Inventory tab UI

Fourth tab ("Inventory") after Validate: three cards — Fonts (user / bundled
lists), Assets (table: path, size), Emoji (slug chips; plugin slugs appear
when the display process is publishing, with a muted note "from the running
display" / "display offline — plugin emoji unknown" when status is absent).

## Component 3: plugin-registered names in status.json

The display process publishes each plugin's real registered names, not just
counts. `_setup_status_board`'s plugins comprehension adds:

```python
"names": dict(getattr(info, "names", None) or {}),
```

(`PluginInfo.names` is the same per-surface dict `led-ticker plugins` prints:
`{"widgets": ["pool.monitor"], "emojis": ["baseball.ball"], ...}`.)

**Schema:** additive field inside existing `plugins[]` entries — top-level key
set unchanged, so `SCHEMA_VERSION` stays 1. The schema tripwire's comment
gains a note that nested additive fields are version-compatible; readers must
tolerate their absence (the page must render counts-only status files from
v1.0 displays without error).

**Page:** the Plugins card shows names grouped by surface (e.g.
`pool — widgets: pool.monitor`), falling back to the counts JSON when `names`
is absent. The Inventory tab's emoji card merges `plugins[].names.emojis`
(and `hires_emojis`) into the slug list, marked as plugin-provided.

## Error handling

- All new endpoints follow v1 rules: degraded states are 200-with-JSON, only
  auth (401) and traversal/unknown-name (404) are HTTP errors; never 500.
- `GET /api/configs` with an unreadable config dir → `{"configs": []}` (200).
- Inventory sections fail independently: a permission error walking assets
  yields `"assets": []`; fonts/emoji still populate. (No `detail` field —
  the page doesn't consume one; empty-and-degraded is the contract.)
- The page's new fetches handle 401 the same way the existing tabs do.

## Testing

- **Traversal suite** (the load-bearing one): the six cases listed under the
  guard, applied to BOTH `/api/validate-file` and `/api/config?name=`.
- Parity: `/api/validate-file` on a file == `validate_config` on its path.
- Inventory: tmp config-dir fixtures (fonts/assets present/absent/unreadable,
  truncation at the 500 cap); emoji lists match the registries; response
  shape pinned.
- Import purity: the existing tripwire must stay green after the registry
  imports — it is the arbiter for how fonts/emoji get enumerated.
- Back-compat: page logic tolerates v1.0 status files without `names`
  (test the merge path with and without).
- Docs drift: config-options unchanged (no new TOML fields — note: v1.1
  adds NO config surface); concepts page gains the two features + the
  redaction caveat, docs-lint clean.

## Out of scope (unchanged from v1)

Any write path (apply/save/restart), live preview, busy toggle. The
"load redacted copy" button edits browser-local text only.
