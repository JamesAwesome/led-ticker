# Design: a "markdown mode" for the docs site (agent-consumable)

**Date:** 2026-06-23
**Status:** Approved for planning

## Motivation

The docs site (`docs/site/`, Astro 6.4 + Starlight 0.40, 55 `.mdx` pages, deployed via Cloudflare/wrangler, now fully public) renders for human browsers. Coding agents consume it poorly: they fetch HTML, fight the Starlight chrome, and burn context. A "markdown mode" exposes the docs as clean Markdown at well-known URLs so an agent can ingest the whole reference (or an index) in one fetch.

## Decision (settled at brainstorm)

Use the **`starlight-llms-txt`** plugin to emit the [llmstxt.org](https://llmstxt.org) outputs at build time. Cloudflare Access is removed, so these endpoints are publicly fetchable by agents with no bypass. Per-page `<url>.md` routes are a deferred follow-up, not v1.

## Architecture

### A. The plugin + outputs

Add `starlight-llms-txt` to the Starlight `plugins: [...]` array in `astro.config.mjs`. On `astro build` it writes static files into `dist/`, served at the site root by the existing Cloudflare static deploy:

- **`/llms.txt`** — the llmstxt.org index: every page titled, linked, and summarized, grouped by docs section.
- **`/llms-full.txt`** — the entire docs concatenated as one clean Markdown document (the "ingest everything" file).
- **`/llms-small.txt`** — a trimmed variant for smaller context windows.

No new infrastructure: a build-time generator on the existing pipeline, regenerated by every `make docs-build`. The plugin is configured (title, description, optional per-section ordering) to match the site.

### B. Component fidelity (the quality crux)

The export is only valuable if the high-signal content survives into Markdown. The most important is the per-widget **`OptionsTable`** field reference (it imports a fact-pack from `docs/content-source/<slug>.md` and renders a Markdown table to HTML via `marked` at build time), the **`TomlExample`** config blocks, and **`RelatedPages`** / **`PluginCatalog`**.

How these survive depends on whether the plugin reads **rendered page content** or **raw MDX source**:

- **If rendered-content based** (likely): the `OptionsTable` HTML `<table>` and `TomlExample` `<pre>` are real HTML at build time and convert back to a Markdown table / fenced code block automatically. No extra work.
- **If raw-MDX based**: those component tags emit nothing in the export. We then add explicit handling so `OptionsTable` inlines its fact-pack `.md` (the raw table) and `TomlExample` becomes a fenced ```toml block — using the plugin's content-transform hook, or a small remark/rehype step in the export path.

The **first plan task determines which path applies** by building with the plugin and inspecting `llms-full.txt` for a known sentinel (the `countdown` widget's field names + its TOML example). Everything downstream is built on that finding — the plan does not assume the happy path.

`DemoGif` has no text an agent can use beyond its `caption`; the export keeps the caption and drops the screenshot/video (agents can't see them — acceptable).

### C. Discoverability

- Serve `/llms.txt` at the **root** (the convention agents and tooling probe).
- Add a small **human-visible pointer** (a footer link or one-line note) so people know to point their agents at it.
- Add a `<link rel="alternate" type="text/markdown" href="/llms.txt">` in the site `<head>` and a `robots.txt` mention, so well-behaved crawlers/agents discover it.

## Data flow

```
astro build
  → starlight-llms-txt walks the docs content
  → emits dist/llms.txt (index), dist/llms-full.txt (all docs, Markdown), dist/llms-small.txt
  → Cloudflare serves them at https://docs.ledticker.dev/llms.txt etc. (public)
Agent: GET /llms.txt (pick pages)  |  GET /llms-full.txt (whole corpus)
Page <head>: <link rel="alternate" type="text/markdown"> + footer pointer + robots.txt
```

## Scope / non-goals

- **IN:** the `starlight-llms-txt` plugin emitting `/llms.txt` + `/llms-full.txt` + `/llms-small.txt` over **all** docs pages (completeness beats curation for agents); verified component fidelity (the `OptionsTable` field tables and `TomlExample` blocks must appear in `llms-full.txt`); root serving + discoverability (footer link, `<link rel>`, robots.txt); a build/CI guard.
- **OUT (follow-up):** per-page `<url>.md` routes; any hand-curation/section-exclusion; content rewriting for agents beyond what the plugin + the OptionsTable/TomlExample fidelity step produce.

## Testing

- **Generation guard:** a check (in `make docs-build` flow or a small test) that after build, `dist/llms.txt`, `dist/llms-full.txt`, and `dist/llms-small.txt` exist and are non-empty.
- **Fidelity sentinel:** the same check asserts `llms-full.txt` contains a known high-value string — a `countdown` widget field name from its `OptionsTable` fact-pack AND a TOML fence — so a future Astro/Starlight/plugin bump that silently guts the component content is caught.
- **No regression:** `make docs-build` and `make docs-lint` stay green; the human site is unchanged.

## Risks

- **Plugin compatibility** with Starlight 0.40 / Astro 6.4 — pin a known-good version; the first task verifies the build succeeds before anything else is built on it.
- **Component-content loss** — the central risk, fully mitigated by the Task-1 inspection + the fidelity sentinel test (we never ship an export that drops the `OptionsTable` reference tables).
- **`llms-full.txt` size** — the whole docs is large but well within what the format intends; `llms-small.txt` + the `/llms.txt` index cover smaller-context agents.
